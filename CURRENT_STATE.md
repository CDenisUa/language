# Current State

_Last updated: 2026-07-15_

## PWA icons

App icons (`public/icons/`) are generated from a single 1254×1254 source artwork. `icon-192`/`icon-512`/`apple-touch-icon` are direct resizes of the source; `icon-512-maskable` is the source scaled to 80% and centered on the app's `#0b0b0f` theme background, satisfying the maskable safe-zone requirement in the global PWA rules.

This is the daily source of truth for what's actually built. If this disagrees with the code, trust the code and fix this file. For what's next, see [`ROADMAP.md`](./ROADMAP.md); for why decisions were made, see [`DECISIONS.md`](./DECISIONS.md).

## Done

- **Project scaffold** — Vite + React 19 + TypeScript, `vite-plugin-pwa` configured (unique manifest `id: /sprachlabor`, icons, apple meta tags, offline app-shell caching via Workbox).
- **Routing shell** — six placeholder pages wired up via `react-router-dom`: Dashboard, Vocabulary, Scheduler, Shadowing, Error Journal, Settings.
- **Two-language theming** — `zustand` store (`useLanguageStore`) drives a `data-language` attribute on the app root; CSS custom property `--color-accent` switches red (German) / blue (English) app-wide.
- **Mobile responsive layout** — `NavBar` is mobile-first: below 768px, the six nav links + locale switch + language switch collapse into a hamburger-triggered dropdown (backdrop, animated icon, closes on link click); at 768px+ a `min-width` media query restores the original horizontal row. The header is `position: sticky` so the toggle stays reachable while scrolling. `ScheduleCalendar`'s week grid scrolls horizontally on narrow screens with a 640px min-width so day headers stay legible, while the toolbar (Today/Back/Next) is deliberately kept outside that scrolling region so it's never out of reach. `ScheduleForm`'s day-of-week picker uses a `repeat(auto-fit, minmax(3.5rem, 1fr))` grid instead of flex-wrap for an even wrap on narrow screens. Verified with Playwright at 320px/390px (no horizontal overflow on any of the six pages) and 1280px (desktop layout unchanged).
- **Branding** — `ChepioTechFooter` component wired in, per global project convention.
- **UI localization (i18n)** — interface text switches between Ukrainian (default) and Russian via `useLocaleStore` (zustand, persisted to `localStorage`), independent of the DE/EN study-language switch. Translation dictionaries live in `src/i18n/translations.ts`; `useTranslation()` is the access hook. Card content (word translations) is a separate, not-yet-built concern.
- **IndexedDB data layer** (`src/services/db/`) — `getDatabase()` singleton via `idb`, schema-typed against `SprachlaborDBSchema` (v2: `words` indexed `by-language`, plus `scheduleEvents`). A generic `createRepository(storeName)` factory gives every store `getAll/getById/put/remove/clear/save`, with `save()` auto-stamping `id`/`createdAt`/`updatedAt` per the sync-readiness convention in `DECISIONS.md` — its type now uses a distributive `Omit` so `save()` still works correctly for discriminated-union stores like `scheduleEvents`. `wordsRepository` adds `getByLanguage()`. `WordRecord` (`src/types/word.ts`) embeds `ts-fsrs`'s own `Card` type directly rather than redefining FSRS fields.
- **Verified in-browser** — dev server, production build, and DE/EN + UA/RU switch behavior manually checked with Playwright.
- **Deployed to GitHub** — `git@github.com:CDenisUa/language.git`, `main` branch.
- **Deployed to Cloudflare Pages** — project `sprachlabor`, production branch `main`, live at https://sprachlabor.pages.dev. `public/_redirects` (`/* /index.html 200`) gives client-side routes a SPA fallback; `public/_headers` sets `Cache-Control: public, max-age=0, must-revalidate` on `/sw.js` so service-worker updates aren't stuck behind stale caching. Deploys are currently manual via `wrangler pages deploy dist --project-name=sprachlabor`; no CI auto-deploy is wired up yet (see `DECISIONS.md`).
- **Vocabulary module** (`src/pages/Vocabulary/`) — full add/edit/delete word CRUD (`WordForm`), an FSRS-driven review queue (`ReviewCard`, one due card at a time, Again/Hard/Good/Easy rating), and a word list (`WordList`) with an Austrian-variant-only filter. New words start immediately due via `createNewCard()`. `src/services/fsrs/fsrsScheduler.ts` wraps `ts-fsrs` (`createNewCard`/`scheduleReview`/`isDue`); `src/services/tts/speak.ts` wraps `SpeechSynthesis` for on-demand pronunciation (front/back/Austrian variant, using `de-DE`/`de-AT`/`en-US` per `src/consts/speechLocales.ts` — nothing is recorded or stored). The Austrian variant field only shows when the active study language is German. Manually verified end-to-end in-browser with Playwright (add → review → rate → filter → language switch).
- **Smart Scheduler** (`src/pages/Scheduler/`) — the `scheduleEvents` store holds recurring rules (`daysOfWeek`/`startTime`/`endTime`) and one-off exceptions (`skip` a recurring date, or `add` a standalone block like the Thursday language exchange). `expandOccurrences()` turns those into concrete calendar occurrences for a date range; `getEnglishSeparationStatus()` is the pure function that says whether English is currently blocked — from the start of today's German block(s) until `ENGLISH_SEPARATION_BUFFER_HOURS` (3h) after the latest one ends. `ScheduleCalendar` renders the current Monday-start week via `react-big-calendar` (red = German, blue = English); `ScheduleForm` adds either kind of event; `SeparationStatus` surfaces the blocked/available state on the page itself — nothing elsewhere in the app reads or enforces it yet (that's for Task 5, Notifications). Manually verified in-browser: a Mon–Thu German block and the Thursday 20:00 "Mi Casa, Graz" exception both render correctly on the calendar and persist across reloads.
- **In-app reminder banner** (`src/components/ReminderBanner/`, Task 5a) — `useReminders()` (`src/hooks/useReminders.ts`) polls the schedule once a minute and surfaces two kinds of dismissible nudges, rendered globally in the `App` shell above `<main>` so they show on every page: an "English unblocked" nudge fired on the blocked->available transition of `getEnglishSeparationStatus()` (`shouldFireUnblockNudge()`), and reminders for any labeled schedule occurrence (recurring or exception) starting within `UPCOMING_EVENT_LEAD_MINUTES` (60) or already in progress (`getUpcomingEventReminders()`) — this is what covers the Thursday "Mi Casa, Graz" exchange, generically rather than as a hardcoded case. Dismissals persist per reminder id in `localStorage` so a closed banner doesn't resurface on reload. Purely a nudge — nothing gates Vocabulary/Shadowing (see `DECISIONS.md`).
- **Web Push** (`worker/`, Task 5b) — a standalone Cloudflare Worker (`sprachlabor-push`, D1-backed, deployed at the URL in `src/consts/push.ts`) with a 5-minute Cron Trigger that reuses `expandOccurrences`/`getEnglishSeparationStatus`/`getUpcomingEventReminders`/`shouldFireUnblockNudge` directly via relative import — no logic duplicated between client and Worker. `zonedNow()` (`worker/src/zonedNow.ts`) compensates for Workers having no local timezone, so that reused local-Date-based logic still reads correctly for Graz (`Europe/Vienna`). Sends via `@block65/webcrypto-web-push` (pure WebCrypto, no Node-crypto dependency). Client side: `usePushSubscription` backs a real toggle on the Settings page (was a placeholder), `usePushScheduleSync` re-syncs current schedule events to the Worker once per app load when subscribed. `src/sw.ts` is now a custom (`injectManifest`-strategy) service worker with `push`/`notificationclick` handlers — `vite-plugin-pwa`'s `generateSW` strategy had no injection point for these. Verified directly against the deployed Worker (CORS preflight, subscribe/schedule routes persisting to D1) and via a real `vite preview` build (service worker registers, Settings toggle reaches the browser's permission prompt) — the notification-permission grant itself and actual push delivery need a manual check on a real device, since this environment's automated browser can't grant that permission (see `DECISIONS.md`).
- **Shadowing Lab** (`src/pages/Shadowing/`, Task 6) — upload your own audio (`TrackUploadForm`, stored as a `Blob` in the new `shadowingTracks` IndexedDB store, schema v3), see your uploaded tracks (`TrackList`), and practice (`ShadowingPractice`): a native `<audio controls>` element plays the track while the user shadows along live — no clip-marking/pause-repeat drill mode (see `DECISIONS.md`). Where `SpeechRecognition` is supported (`src/services/speechRecognition/`, Chrome/Edge/Android; feature-detected via `isSpeechRecognitionSupported()`), live recognized speech is shown as in-the-moment feedback only ("Heard: ..."), never diffed against a transcript — there is no transcript field and no computed accuracy score. Safari/iOS gets a plain "not supported, rate yourself" message instead, matching the project's existing WebKit `SpeechRecognition` gap. Every session, regardless of browser, is scored by the user's own 1–5 self-rating plus optional notes; `practiceDurationSeconds` is read from `audio.currentTime` at save time. The user's own voice is never recorded or stored — mic input is used live via `SpeechRecognition` and discarded per result. Sessions persist to the new `shadowingSessions` store (`trackId`/`language`/`rating`/`practiceDurationSeconds`/`notes`). Manually verified end-to-end in-browser with Playwright (upload a real WAV → practice view loads it with a correct blob URL and duration → live "Heard:" feedback panel renders in Chromium → rate → save → session round-trips through IndexedDB with the correct rating → delete track) and in both uk/ru locales.
- **Error Journal** (`src/pages/ErrorJournal/`, Task 7) — manual mistake logging: `ErrorEntryForm` (mistake + correction + category — grammar/vocabulary/pronunciation/other — + optional note + an optional link to one existing Vocabulary word or Shadowing track, picked from a combined `<select>` built from `useWords`/`useShadowingTracks` for the active language), `ErrorEntryList` (shows a category badge, the mistake struck through → correction, note, and a resolved `🔗 <label>` line when linked; edit/delete like Vocabulary). The link is stored as a plain `linkedRecordType`/`linkedRecordId` pair (`src/types/errorJournalEntry.ts`) — no foreign-key enforcement, resolved back to a display label at render time by matching against the current word/track list, so a since-deleted linked record just stops showing a link rather than erroring. New IndexedDB store (schema v4): `errorJournalEntries`, indexed `by-language`. Manually verified end-to-end in-browser with Playwright (added a Vocabulary word → confirmed it appeared in the link picker → added a linked entry → edit pre-filled every field including the link and category → delete → empty state) in the ru locale.
- **Dashboard analytics** (`src/pages/Dashboard/`, Task 8) — `useStudyTimer(language)` (`src/hooks/useStudyTimer.ts`) is wired into the Vocabulary page and automatically logs study time: it accrues while the page is mounted and the tab is visible, and — critically — persists to the new `vocabularyStudySessions` store (schema v5) *immediately on every `visibilitychange` to hidden*, not only on React's unmount cleanup. This was a deliberate fix after Playwright verification showed a real gap: a hard navigation or tab close never runs an unmount cleanup, so an earlier "accumulate locally, save on unmount" version silently lost that session's time. Sessions shorter than `MIN_STUDY_SESSION_SECONDS` (5) aren't saved, to filter out accidental page visits. Shadowing already tracks its own `practiceDurationSeconds` per session (Task 6), so Dashboard sums both sources by language rather than re-instrumenting Shadowing. Scheduler's planned blocks are deliberately excluded — only actual logged activity counts (see `DECISIONS.md`). `StudyHours` shows hours per language plus actual-vs-target balance; the target is a configurable Settings value (`useStudyBalanceStore`, zustand+`localStorage`-persisted like the locale/language switches, default 70% German — new `settings-balance` section on the Settings page, clamped 0–100). `ErrorBreakdown` groups Task 7's `errorJournalEntries` by category, reusing the same category labels rather than duplicating translation strings. Manually verified end-to-end in-browser with Playwright: changed the Settings target to 65%, spent real time on the Vocabulary page, used a client-side nav-link transition (not `page.goto()`, which is a hard reload and doesn't run React cleanup — confirmed this distinction matters) to trigger the session save, confirmed the exact duration via a direct IndexedDB read, added Error Journal entries in two categories, and confirmed both the hours/balance and the category breakdown rendered correctly on Dashboard in both uk/ru locales.
- **Export/Import JSON backup** (`src/services/backup/backupService.ts`, Task 9) — a new "Резервна копія" section on Settings. `createBackup()` gathers `words`/`scheduleEvents`/`shadowingSessions`/`errorJournalEntries`/`vocabularyStudySessions` in full via each repository's `getAll()`, plus `shadowingTracks` as metadata only (title/language/mimeType/timestamps — the audio `Blob` itself is stripped out before serializing, since JSON can't hold binary data and re-encoding audio as base64 was rejected as unnecessary bloat, see `DECISIONS.md`). `downloadBackup()` triggers a real browser file download (`sprachlabor-backup-<date>.json`) via an object URL and a temporary anchor click — verified with Playwright to produce an actual downloaded file with the expected shape. Import is file-picker → `isValidBackupFile()` structural check → an explicit confirm step (`pendingImport` state, nothing is touched until the user clicks confirm) → `restoreBackup()`, which **clears and replaces** `words`/`scheduleEvents`/`shadowingSessions`/`errorJournalEntries`/`vocabularyStudySessions`, but deliberately leaves the `shadowingTracks` store completely untouched (existing audio survives an import intact, since the backup never had audio to restore it with anyway) and reports how many shadowing-track entries from the file were skipped. A real gotcha handled here: `WordRecord.fsrs.due`/`fsrs.last_review` are `Date` objects that JSON serializes to strings but does *not* revive back into `Date`s on parse — `restoreBackup()`'s `reviveWordDates()` reconstructs them explicitly, otherwise `isDue()` would throw on the very next Vocabulary page visit. After a successful restore the UI shows a manual "reload" button rather than auto-reloading, since every page's data hooks (`useWords`, `useScheduleEvents`, etc.) have already loaded stale in-memory state that a silent background restore wouldn't refresh. Manually verified end-to-end in-browser with Playwright: exported a real backup (word + 2 schedule events) → downloaded and inspected the actual JSON file → edited it → re-imported → confirmed the confirm-step gate, the restored word (renamed) round-tripped correctly *and* was still due for FSRS review (proving the Date revival works on a real browser JSON round-trip, not just a mocked unit test) → confirmed both schedule events survived → confirmed the invalid-file error path with a non-JSON file.
- **Text Analyzer** (`src/pages/TextAnalyzer/`, `/text-analyzer`, 7th nav item) — paste text in the active study language into a textarea and click "Аналізувати" to get it back with advanced words highlighted inline. `src/services/frequency/tokenize.ts` is a pure word/non-word segmenter (`[a-zA-ZäöüßÄÖÜ']+`, keeps German umlauts/ß and English apostrophes as part of a word, and reproduces the original text exactly when segments are rejoined — no lemmatization needed since the frequency data is built from real inflected usage, not dictionary lemmas). `src/services/frequency/frequencyLevels.ts` fetches and memoizes a word→rank `Map` per language from two static datasets (`public/data/frequency-{de,en}.json`, ~500KB each, 50,000 words ranked by frequency) and buckets a rank into a rough CEFR-like tier via `getDifficultyTier()` — `common` (rank ≤3000, rendered as plain text) / `B2` (≤6000) / `C1` (≤15000) / `C2` (≤50000) / `unranked` (not in the top 50k at all — likely a proper noun, typo, or genuinely rare word). `src/services/frequency/analyzeText.ts` composes the two. This is a frequency-rank approximation, not a real CEFR wordlist — see `DECISIONS.md` for why (no open, full-range German CEFR list exists, so the same frequency-based method is used uniformly for both languages instead of a stronger-but-asymmetric approach). Data: `FrequencyWords` by Hermit Dave (OpenSubtitles2018 corpus), CC BY-SA 4.0 — credited both in a footer note on the page (both locales) and in `DECISIONS.md`. Clicking a flagged word opens `WordForm` (reused directly from Vocabulary, now takes an optional `initialFront` prop to prefill without an `editingWord`) to add it to the vocabulary; words already saved render as a checked, non-clickable `✓` badge instead, computed reactively from `useWords()` so a save immediately updates the highlighting without re-analyzing. The two frequency JSON files are precached by the service worker (added `json` to `vite.config.ts`'s `injectManifest.globPatterns`) — confirmed via a real `vite preview` + Playwright `caches` inspection that both land in the Workbox precache, so the feature works fully offline after first install. Manually verified end-to-end in-browser with Playwright using real German and English paragraphs: results were qualitatively sensible (e.g. German "Wirtschaft"/"Herausforderungen" flagged C1, "Nachhaltigkeit"/"verantwortungsvollem" unranked, "Ich gehe heute nach Hause und trinke Kaffee" left entirely unstyled; English "entrepreneurs"/"resilience" flagged C2 while "I want to eat lunch now" stayed plain) — added a flagged word to Vocabulary and confirmed it appeared correctly in the FSRS review queue, and checked both uk/ru locales.
- **Test infra** — 116 app tests across 33 files, plus 3 Worker tests (`worker/`, its own `vitest.config.ts`, deliberately excluded from the app's vitest run). `src/setupTests.ts` polyfills jsdom gaps: `localStorage`, IndexedDB (`fake-indexeddb/auto`), `SpeechSynthesisUtterance`, and swaps jsdom's `Blob`/`File` for Node's native `node:buffer` versions, since jsdom's own classes silently lose their content under `structuredClone` (which `fake-indexeddb` relies on for every `put()`); see `DECISIONS.md`. All new UI text across every module is localized (uk/ru) through the i18n system.

## Known constraints carried forward

- No speech-to-text on iOS/Safari — Vocabulary's pronunciation stays TTS-only there, and Shadowing Lab's live "Heard: ..." feedback simply isn't shown (self-rating still works) — see `DECISIONS.md`.
- App data (words, review history, schedule) is still local-first with no cross-device sync. The Task 5b Worker/D1 is narrowly scoped to push subscriptions and a schedule-events snapshot for reminder-scheduling — not a general sync layer.
